## Bootstrap

```js
import { component, html, tutuca } from "tutuca";

const Counter = component({
  name: "Counter",
  fields: { count: 0 },
  methods: {
    inc() {
      return this.setCount(this.count + 1);
    },
  },
  view: html`<button @on.click=".inc" @text=".count"></button>`,
});

const app = tutuca("#app");
app.registerComponents([Counter]);
app.state.set(Counter.make({}));
app.start();
```

The app exposes a few runtime hooks:

- `app.onChange((info) => ...)` — fires after every state change with
  `{ val, old, info, timestamp }`. Useful for logging or persisting state.
- `app.stop()` — removes all event listeners and cancels cache eviction.
  Pair with `app.start()` for teardown in tests or SPA navigation.

## Notation Reference

| Prefix   | Means                                     | Example               |
| -------- | ----------------------------------------- | --------------------- |
| `.x`     | field or method on `this`                 | `.count`, `.inc`      |
| `@x`     | local binding (loop / scope)              | `@key`, `@value`      |
| `^x`     | macro parameter                           | `^label`              |
| `$x`     | computed property (cached)                | `$totalChars`         |
| `!x`     | request handler                           | `!loadData`           |
| `*x`     | dynamic binding                           | `*theme`              |
| `Name`   | component type (PascalCase)               | `Item`, `JsonNull`    |
| `name`   | bare identifier — meaning depends on slot | `dec`, `value`, `ctx` |
| `'str'`  | string literal                            | `'btn btn-success'`   |
| `{expr}` | interpolation in attr text                | `Hi {.name}`          |
| `.s[.k]` | sequence/map item access                  | `.byKey[.currentKey]` |

A bare `name` (no prefix) resolves differently depending on where it appears
in an event handler `@on.<event>="<handler> <arg> <arg> ..."`:

- **First slot** — the input/alter handler name (e.g. `dec`, `filterItem`).
  Equivalent to a method, but looked up in `input` / `alter` instead of
  `methods`. Use `.name` for `methods`.
- **Subsequent slots** — a built-in handler arg name resolved by
  `transactor.js` `lookupName`: `value`, `valueAsInt`, `valueAsFloat`,
  `target`, `event`, `isAlt`, `isShift`, `isCtrl`/`isCmd`, `key`,
  `keyCode`, `isUpKey`, `isDownKey`, `isSend`, `isCancel`, `isTabKey`,
  `ctx`, `dragInfo`. Anything else triggers a warning at compile time.

```html
<button @on.click="addItem JsonSelector ctx">+</button>
<!--                ↑ handler ↑ Type      ↑ built-in arg -->
```

## Quoting & String Literals

Tutuca's expression parser is context-sensitive: where `:attr=` and
interpolation `{...}` accept string templates, `@if`, `@each`, `<x render=>`
do not. The rules:

| Form                | Example                | Where it works                                   |
| ------------------- | ---------------------- | ------------------------------------------------ |
| `'string'`          | `@then="'btn ok'"`     | anywhere a value is allowed                      |
| Bare with `{...}`   | `:class="btn {.kind}"` | `:attr=`, `@text`, `@title`, macro dynamic attrs |
| Bare without quotes | `flex gap-3`           | **never** — returns `null`                       |
| Bare identifier     | `dec`, `value`         | name slots only (handler/arg, not as a value)    |

Why: outside braces, an unquoted multi-word string fails the value parser.
Inside `{...}`, every part is parsed individually so `flex gap-3 {.color}`
works without quotes, but the surrounding text needs interpolation present
to enable the string-template path.

```html
<!-- ✅ all valid -->
<p :class="'flex gap-3'">x</p>
<p :class="flex {.color}">x</p>
<!-- {…} enables template -->
<p :class="static-classes {''}">x</p>
<!-- folds to a const -->

<!-- ❌ -->
<p :class="flex gap-3">x</p>
<!-- null: no quotes, no braces -->
<x render="'foo bar'"></x>
<!-- @render rejects string templates -->
```

## Component Skeleton

```js
component({
  name: "MyComp",
  fields: {                    // see "Field Types"
    count: 0,
    items: [],
    nullable: null,
  },
  view: html`...`,             // default view (named "main")
  views: {                     // additional views
    edit: html`...`,
    big: { view: html`...`, style: css`.x { color: red }` },
  },
  style: css`...`,             // scoped to main view
  commonStyle: css`...`,       // scoped to all views of this component
  globalStyle: css`...`,       // injected globally, no scoping
  methods: { ... },            // called via `.name`
  input: { ... },              // called via `name` (no dot)
  alter: { ... },              // for @when / @enrich-with / @loop-with
  computed: { ... },           // referenced as $name, cached per instance
  logic: { init(ctx) { ... } },// called explicitly
  bubble: { name(arg, ctx) { ... } },   // events bubbling up
  response: { name(res, err) { ... } }, // async request results
  statics: { fromData(d) { return this.make(...) } },
  dynamic: { theme: ".color" },
  on: { stackEnter() { return []; } }, // only implement it if using dynamic bindings
});
```

`Comp.make({...})` builds an instance. Coercion is automatic: arrays become
`List`, plain objects become `IMap`, native `Set` becomes `ISet` from immutablejs.

## Field Types & Auto-generated API

`fields: { name: defaultValue }` — type inferred from the default.

| Default              | Field type | Auto-generated methods (for field `x`)                                                                                             |
| -------------------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `"hi"`               | text       | `setX`, `updateX`, `resetX`, `isXSet`, `isXNotSet`, `isXEmpty`, `xLen`                                                             |
| `42`                 | float      | `setX`, `updateX`, `resetX`, `isXSet`, `isXNotSet`                                                                                 |
| `true`               | bool       | `setX`, `toggleX`, `updateX`, `resetX`, `isXSet`, `isXNotSet`                                                                      |
| `null`               | any        | `setX`, `updateX`, `resetX`, `isXSet`, `isXNotSet`                                                                                 |
| `[]`/`List()`        | list       | `setX`, `pushInX`, `insertInXAt`, `setInXAt`, `getInXAt`, `updateInXAt`, `deleteInXAt`/`removeInXAt`, `isXEmpty`, `xLen`, `resetX` |
| `{}`/`IMap()`        | map        | `setInXAt`, `getInXAt`, `updateInXAt`, `deleteInXAt`, `isXEmpty`, `xLen`, `resetX`                                                 |
| `OMap()`             | omap       | same as map (preserves insertion order)                                                                                            |
| `ISet()`/`new Set()` | set        | `addInX`, `deleteInX`, `hasInX`, `toggleInX`, `xLen`, `isXEmpty`, `resetX`                                                         |
| `KList`              | KList      | (extra build) `pushInX`, `setInXAt`, `getInXAt`, `deleteInXAt`, `moveKeyBeforeKey`/`AfterKey`                                      |

Explicit field types via `classFromData`:

```js
fields: {
  count: { type: "int", defaultValue: 10 },     // text/int/float/bool/list/map/omap/set/any
  child: { component: "Item", args: { ... } },  // nested component instance
}
```

## Statics

`statics: {...}` adds methods to the component **class** itself (not to
instances). They become available as `Comp.Class.<name>(...)`, alongside
the auto-generated `Comp.Class.make(...)` (which `Comp.make(...)` aliases).

Most common use: a `fromData` factory that builds an instance from plain
JS data (e.g. JSON loaded from disk), recursively constructing children:

```js
statics: {
  fromData({ items = [] }) {
    return this.make({ items: items.map((v) => Item.Class.fromData(v)) });
  },
}
// usage: TreeRoot.Class.fromData([...])
```

Inside a static, `this` is the class itself, so `this.make(...)` calls the
auto-generated constructor.

## Text Rendering

```html
<span @text=".str"></span>
<!-- prepend text into span -->
<x text=".bool"></x>
<!-- text-only, no DOM element -->
<x text=".getStrUpper"></x>
<!-- methods are called -->
<x text="$totalChars"></x>
<!-- computed -->
<x text="@value"></x>
<!-- loop binding -->
```

## Attribute Binding

```html
<input :value=".str" @on.input=".setStr value" />
<a :href=".url" :title="Hi {.name}">link</a>
<!-- string template -->
<button class="btn" :class="btn {.color}">x</button>
```

Plain attrs are static. `:attr="..."` is a dynamic expression. Boolean HTML
attributes (`disabled`, `checked`, `hidden`, …) are auto-recognized; pass a
boolean field.

SVG attributes are case-sensitive but the HTML parser lowercases them.
Tutuca special-cases `:viewbox` → `viewBox` so SVG roots work with the
expected casing; for other camelCased SVG attrs, use `setAttributeNS`-aware
elements or wrap them in components that emit raw markup.

## Event Handling

```html
<!-- method (`.`) vs input handler (no dot) -->
<button @on.click=".inc">+</button>
<button @on.click="dec">-</button>

<!-- pass args by name -->
<input @on.input=".setStr value" />
<input @on.input=".setN valueAsInt" />
<button @on.click=".pick @key isAlt">pick</button>
<button @on.click=".addItem JsonSelector">+</button>
<!-- type as arg -->
<button @on.click=".loadAnotherWay ctx">load</button>
<!-- ctx -->
```

Built-in handler arg names: `value`, `valueAsInt`, `valueAsFloat`, `target`,
`event`, `isAlt`, `isShift`, `isCtrl`/`isCmd`, `key`, `keyCode`, `isUpKey`,
`isDownKey`, `isSend`, `isCancel`, `isTabKey`, `ctx`, `dragInfo`.

### Event Modifiers

`@on.<event>+<mod>+<mod>=...`

- All events: `+ctrl`, `+cmd`/`+meta`, `+alt`
- `keydown` only: `+send` (Enter), `+cancel` (Escape)

```html
<input @on.keydown+send=".submit value" @on.keydown+cancel=".reset" />
<button @on.click+ctrl=".soloOnly">ctrl-click</button>
```

### Web Components & Custom Events

Custom elements just work, and any `CustomEvent` they fire is reachable via
`@on.<event-name>`. The event's `detail` surfaces as `value`:

```js
import "https://esm.sh/emoji-picker-element";

input: { onPick(detail) { return this.setCurrent(detail.unicode); } }
view: html`
  <emoji-picker @on.emoji-click="onPick value"></emoji-picker>
`,
```

## Conditional Display

```html
<div @show=".isLoading">Loading...</div>
<div @hide=".isLoading">content</div>

<!-- Single @if: shorthand @then/@else (attr inferred) -->
<button @if.class=".isActive" @then="'btn btn-success'" @else="'btn btn-ghost'">
  ...
</button>

<!-- Multiple @if on same element: name the attr explicitly -->
<button
  @if.class=".isActive"
  @then="'on'"
  @else="'off'"
  @if.title=".isActive"
  @then.title="'On'"
  @else.title="'Off'"
>
  ...
</button>
```

> HTML disallows two attributes with the same name on the same element. So
> when there are multiple `@if.<attr>` directives on the same element, every
> `@then` and `@else` after the first **must** include the attribute name
> (`@then.title`, `@else.title`) — otherwise the second `@then=`/`@else=`
> is dropped by the HTML parser before tutuca ever sees it, and there's no
> way to recover which `@if` it was meant to pair with.

## List Iteration

```html
<!-- iterate plain values -->
<li @each=".items"><span @text="@key"></span>: <x text="@value"></x></li>

<!-- filter -->
<li @each=".items" @when="filterItem">...</li>

<!-- per-item enrichment (binds.X => @X in template) -->
<li @each=".items" @enrich-with="enrichItem">
  <x text="@count"></x>
</li>

<!-- shared per-loop data (computed once before iteration) -->
<li @each=".items" @loop-with="getIterData" @when="filterItem">...</li>

<!-- render a list of components -->
<x render-each=".items"></x>
<x render-each=".items" as="edit"></x>
<!-- each item rendered with the "edit" view -->
<x render-each=".items" when="filterItem"></x>
<!-- with filter -->
<x render-each=".items" loop-with="getIterData" when="filterItem"></x>
<!-- with shared loop data -->

```

```js
alter: {
  filterItem(_key, item, iterData) { return item.includes(iterData.q); },
  enrichItem(binds, _key, item, iterData) { binds.count = item.length; },
  getIterData(seq) { return { q: this.query.toLowerCase() }; },
}
```

### Lifecycle of `@each`

For each render of an element with `@each=".items"`:

1. **Resolve sequence** — evaluate `.items`. Lists, IMaps, OMaps, ISets, and
   any class registered in `seqInfoByClass` (e.g. `KList`) are recognized.
2. **`@loop-with`** (once per render) — `getIterData.call(this, seq)` is
   called and its return value becomes `iterData`. Skipped if no
   `@loop-with`; in that case `iterData` is `{ seq }` from the default.
3. For each `(key, value)` pair in the sequence:
   1. **`@when`** — `filterItem.call(this, key, value, iterData)`; if it
      returns `false`, the item is skipped.
   2. **`@enrich-with`** — `enrichItem.call(this, binds, key, value, iterData)`.
      `binds` is a **mutable object** seeded with `{ key, value }`; mutating
      it (`binds.count = ...`) creates `@`-prefixed bindings available in
      the templated children. The return value is ignored.
   3. **Render** the element with the new bindings on the stack.

The auto-bound names inside the loop are always `@key` and `@value` (or
whatever you wrote into `binds`).

### Handler resolution for loop directives

`@when` / `@enrich-with` / `@loop-with` accept the same forms as event
handler names:

- `@when="filterItem"` — looks up `alter.filterItem` (idiomatic).
- `@when=".filterItem"` — looks up a method on `this` instead. **Works,
  but not idiomatic** — `alter` exists to keep iteration helpers grouped
  separately from public methods.

### `@enrich-with` outside `@each`

Without an `@each` on the same element, `@enrich-with` becomes a scope
enricher (see _Scope Enrichment_ below). The function takes no `binds` arg;
its **return value** is the bindings object.

## Scope Enrichment

`@enrich-with` on a non-`@each` element returns an object whose keys become
`@`-prefixed bindings for descendants:

```js
alter: { enrichScope() { return { len: this.text.length }; } }
```

```html
<div @enrich-with="enrichScope">Length: <x text="@len"></x></div>
```

## Computed Properties

```js
computed: {
  totalItemsChars() {
    let n = 0;
    for (const item of this.items) n += item.length;
    return n;
  },
}
```

```html
<x text="$totalItemsChars"></x>
<!-- cached per instance -->
```

The `$` prefix marks any computed reference (template attr, text, handler
arg). See *Notation Reference*.

## Rendering Components

```html
<x render=".item"></x>
<!-- default ("main") view -->
<x render=".item" as="edit"></x>
<!-- specific view -->
<x render-it></x>
<!-- only inside @each / render-each -->
<x render=".byIndex[.currentIndex]"></x>
<!-- list item access -->
<x render=".byKey[.currentKey]"></x>
<!-- map item access -->
```

### `as` and view names

A component's top-level `view` field is registered under the name `"main"`
and is the default. Any extra views go under `views: { name: html\`...\` }`.

`<x render=".item" as="edit">` selects the `edit` view of `.item`. If the
component has no `edit` view, tutuca falls back to `main`. `as` only
applies to the **direct** component being rendered — descendants render
under whatever they would normally pick. Use `@push-view` (below) to
influence rendering of the whole subtree.

## Multiple Views & View Stack

```js
component({
  view: html`<p @text=".title"></p>`, // "main"
  views: {
    edit: html`<input :value=".title" @on.input=".setTitle value" />`,
  },
});
```

```html
<!-- @push-view pushes a name onto the rendering stack;
     descendants resolve to first matching view, falling back to "main" -->
<div @push-view=".view"><x render-each=".items"></x></div>
```

`@push-view` differs from `as` in scope:

- `as="edit"` only changes the view of the **one** `<x render>` element
  it's attached to.
- `@push-view=".view"` pushes a name onto the rendering stack that applies
  to **every** component rendered recursively under it — direct children
  _and_ their descendants. Each component picks the first stack entry it
  has a matching view for, falling back to `"main"`. Stack pushes nest, so
  inner `@push-view` directives extend (not replace) the outer ones.

## Styles

```js
component({
  style: css`
    .mine {
      color: red;
    }
  `, // scoped to main view
  commonStyle: css`
    .shared {
      color: yellow;
    }
  `, // scoped to all views of this component
  globalStyle: css`

    }
  `, // global, no scoping

        }
      `,
    },
  },

```

Tagged templates `html` and `css` are just `String.raw` (editor hinting only).
Plain strings work too.

## Bubble Events

```js
input: { onClick(ctx) { ctx.bubble("treeItemSelected", [this]); return this; } },
bubble: {
  treeItemSelected(selected, ctx) {
    // ctx.stopPropagation() to halt
    return this.insertInLogAt(0, `selected ${selected.label}`);
  },
}
```

`ctx.bubble("name", args)` walks the path used to dispatch the originating
event back toward the root of the state tree. At each ancestor instance,
tutuca calls `bubble.<name>(...args, ctx)` if the ancestor's component
defines one — components without a matching handler are skipped silently.
Bubbling continues until either:

- the root state value is reached, or
- a handler calls `ctx.stopPropagation()`.

Each handler returns a (possibly updated) instance of itself, just like
`input` / `methods` / `logic`. Because each ancestor sees the event
_after_ descendants have already been transacted, you can use bubble
handlers to maintain aggregate state (logs, selections, totals).

## Logic / Lifecycle

`logic` is invoked explicitly by app code or by another handler — there is
**no built-in lifecycle**. `logic.init` is just a convention; nothing in
tutuca calls it automatically. The host application must dispatch it (for
example after `app.start()`) for it to run.

```js
logic: { init(ctx) { ctx.request("loadData"); return this.setIsLoading(true); } }
```

Dispatch from anywhere:

```js
app.dispatchLogicAtRoot("init"); // host code, top-level
ctx.at.field("personalSite").logic("init"); // child by field name
ctx.at.index("items", 3).logic("init"); // list element at index 3
ctx.at.key("byKey", "k1").logic("init"); // map / KList entry by key
ctx.at.field("a").field("b").index("xs", 0).logic("ping"); // chain freely
ctx.logic("name"); // self
ctx.bubble("name", [arg]); // bubble up
```

`ctx.at` returns a `PathBuilder` with `.field(name)`, `.index(name, i)` and
`.key(name, k)`. Each call appends a step to the dispatcher's current path
before `.logic(...)` / `.bubble(...)` fires, so the handler runs inside the
child instance with `this` bound to it.

## Async Requests

```js
export function getRequestHandlers() {
  return {
    async loadData() {
      const r = await fetch("https://example.com/data.json");
      return await r.json();
    },
  };
}

// register at the same scope where you registerComponents
const scope = app.registerComponents([Comp]);
scope.registerRequestHandlers(getRequestHandlers());
```

In a component:

```js
logic:    { init(ctx) { ctx.request("loadData"); return this.setIsLoading(true); } },
response: { loadData(res, err) { return this.setIsLoading(false).setItems(res); } },
methods:  {
  loadAlt(ctx) {
    ctx.request("loadData", [], { onOkName: "loadDataOk", onErrorName: "loadDataErr" });
    return this.setIsLoading(true);
  },
}
```

## Drag and Drop

```html
<div
  @each=".items"
  draggable="true"
  data-dragtype="my-item"
  data-droptarget="my-item"
  @on.drop="onDrop @key dragInfo event"
></div>
```

```js
input: {
  onDrop(targetKey, dragInfo, e) {
    const sourceKey = dragInfo.lookupBind("key");   // any bind from source render
    return this.setItems(this.items.moveKeyBeforeKey(sourceKey, targetKey));
  },
}
```

Tutuca auto-manages two attrs during a drag — style them with CSS:

```css
[data-dragging="1"] {
  opacity: 0.5;
}
[data-draggingover="my-item"] {
  outline: 1px dashed;
}
```

Touch is wired up too (drag is fired after a small move threshold).

## Macros

Pure template expansion — no state, no methods. Calls inside a macro resolve
against the _host_ component.

```js
import { macro, html } from "tutuca";

const badge = macro(
  { label: "'New'", kind: "'info'" }, // defaults are *expressions*
  html`<span :class="badge badge-{^kind}" @text="^label"></span>`,
);

export function getMacros() {
  return { badge };
}
```

```html
<x:badge></x:badge>
<!-- defaults -->
<x:badge label="Sale"></x:badge>
<!-- static string (no quotes needed) -->
<x:badge :label="'Sale'"></x:badge>
<!-- dynamic literal -->
<x:badge :label=".status"></x:badge>
<!-- field reference -->
```

Register macros at the same scope as components:

```js
const scope = app.registerComponents([Comp]);
scope.registerMacros(getMacros());
```

```js
const card = macro(
```

```js
const panel = macro(
```


## Raw HTML (escape hatch)

```html
<div @dangerouslysetinnerhtml=".trustedHtml"></div>
```

Bypasses all escaping. Children of the element are ignored when active.

## Immutable Re-exports

`tutuca` re-exports **everything** from
[`immutable`](https://immutable-js.com/) (so `Map`, `Set`, `List`,
`OrderedMap`, `Record`, `Seq`, `is`, `fromJS`, ... are all available),
plus a few short aliases on top to avoid name clashes with the host
runtime's `Map` and `Set`:

```js
import {
  IMap, // alias for immutable Map
  OMap, // alias for OrderedMap
  ISet, // alias for Set
  isIMap, // alias for isMap
  isOMap, // alias for isOrderedMap
  List, // re-exported as-is
  Record, // re-exported as-is
  Seq,
  fromJS,
  is, // ...and everything else immutable exports
} from "tutuca";
```

## Tailwind / MargaUI Class Compilation (extra build)

```js
import { compileClassesToStyleText, injectCss, tutuca } from "tutuca/extra";
import { compile } from "https://esm.sh/margaui";

const app = tutuca("#app");
app.registerComponents([Comp]);
const css = await compileClassesToStyleText(app, compile);
injectCss("myapp", css);
app.start();
```

## KList (extra build)

Ordered, keyed, immutable collection — like a `Map` whose insertion order you
can rearrange:

```js
import { KList } from "tutuca/extra";

let kl = new KList();
kl = kl.push("a"); // auto key
kl = kl.set("custom", "b");
kl = kl.moveKeyBeforeKey("custom", kl.order.first());
kl = kl.delete("custom");
kl.size;
kl.get(k);
kl.toJS();
```

Custom seq types: `KList` registers itself with `seqInfoByClass` so `@each`
walks it correctly. To make `@each` recognize your own collection class do
the same:

```js
import { seqInfoByClass } from "tutuca";

seqInfoByClass.set(MyClass, (seq, visit) => {
  for (const [k, v] of seq.entries()) visit(k, v, "data-sk");
});
```

## Conventional Module Exports

Examples and the storybook glue follow this shape so files compose freely
and the CLI (next section) can introspect any module without per-app glue:

```js
export function getComponents()       { return [Comp, ...]; }
export function getMacros()           { return { name: macro }; }     // optional
export function getRequestHandlers()  { return { name: async fn }; }   // optional
export function getRoot()             { return Root.make({...}); }
export function getExamples()         {
  // Return one section, or an array of sections.
  return {
    title: "...",
    description: "...",
    items: [{ title, description, value, view }],         // value = Comp.make(...)
  };
}
```

## CLI (`tutuca`)

Inspect, document, lint and render component modules from the command line.
The CLI ships in the package and works on any module that follows the
_Conventional Module Exports_ above.

### Install / invoke

```sh
# project local
npm i --save-dev tutuca
npx tutuca <module-path> <command> [name] [flags]

# global
npm i -g tutuca
tutuca <module-path> <command> [name] [flags]

# from a checkout of this repo
bun tools/tutuca.js <module-path> <command> [name] [flags]
```

The module path comes **first**, the command second, an optional component
name third. `tutuca help` prints the full reference; `tutuca help <command>`
prints a one-liner. `tutuca` ↔ `tutuca -h` prints overview.

### Commands

| Command         | Purpose                                                                                                                |
| --------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `info`          | Summarize which `getX()` exports are present and counts                                                                |
| `list`          | List components with their views and fields (name + type)                                                              |
| `examples`      | Print `getExamples()` content (title, items, per section)                                                              |
| `docs [name]`   | Generate API docs (methods, input handlers, fields with auto-generated accessors) — all or one                         |
| `lint [name]`   | Run the linter; exits **2** on any error-level finding                                                                 |
| `render [name]` | Render examples to HTML in a headless DOM. Filter by component name or `--title`/`--view`. Exits **3** on render crash |
| `doctor`        | `lint` + `render` in one pass — the smoke-test invocation for CI                                                       |
| `help [cmd]`    | Show usage; the only command that does **not** need a module path                                                      |

### Global flags

```
-f, --format <cli|md|json|html>  output format
                                 defaults: info/list/examples/lint/doctor → cli
                                           docs/render                    → md
                                 html only valid for render
                                 json works for every command
-o, --output <file>              write to file instead of stdout
    --pretty                     pretty-print HTML (md/html) via prettier;
                                 JSON formatter uses indent 2
    --module <path>              alternative to first-positional module path
-h, --help                       show help (overview, or for one command)
```

### Exit codes

| Code | Meaning                                                  |
| ---- | -------------------------------------------------------- |
| `0`  | success                                                  |
| `1`  | usage error (bad args, missing module, bad module shape) |
| `2`  | lint findings at error level                             |
| `3`  | render crash                                             |

### Examples

```sh
tutuca ./src/components.js info                       # quick overview
tutuca ./src/components.js list                       # components, views, fields
tutuca ./src/components.js docs Button -f json        # one component, JSON
tutuca ./src/components.js render -f html --pretty -o out/examples.html
tutuca ./src/components.js render Button --title "Disabled state"
tutuca ./src/components.js doctor                     # CI smoke test
```

## Advanced

### Dynamic Bindings

For passing values "context-style" through nested components without prop
drilling. Define on the producer; alias on consumers; resolve as `*name`.

```js
const Theme = component({
  name: "Theme",
  fields: { color: "blue" },
  dynamic: { color: ".color" },
  on: {
    stackEnter() {
      return ["color"];
    },
  },
});
const Child = component({
  dynamic: { color: { for: "Theme.color", default: "'gray'" } },
  view: html`<p :style="color: {*color}"></p>`,
});
```

### Pseudo-`x` (`@x`)

Tutuca's special operations (`render`, `render-it`, `render-each`, `text`,
`show`, `hide`, `slot`) live on the `<x>` tag. That works almost
everywhere, but the browser's HTML parser refuses to keep `<x>` (or any
unknown tag) as a child of certain elements: `<select>` only allows
`<option>` / `<optgroup>`, `<table>` only allows `<thead>` / `<tbody>` /
`<tr>`, `<tr>` only allows `<th>` / `<td>`, etc. Drop `<x render-each>`
inside one of those and the parser silently strips it.

The escape hatch: prefix the **first** attribute on a _legal_ tag with
`@x`. Tutuca treats that tag as if it were `<x>` and reads the next
attribute as the special op.

```html
<!-- ❌ <x> stripped by the HTML parser inside <select> -->
<select>
  <x render-each=".items" as="option"></x>
</select>

<!-- ✅ pseudo-x: <option @x render-each=".items" as="option"> -->
<select>
  <option @x render-each=".items" as="option"></option>
</select>
```

Notes:

- `@x` must be the **first** attribute; the special op (`render-each`,
  `render`, `text`, `show`, ...) is the second.
- The host tag (here `<option>`) is otherwise ignored — only the special
  op runs. Tutuca will produce the rendered children directly.
- Same trick works inside `<tr>`, `<table>`, `<colgroup>`, `<dl>`,
  `<details>`, or anywhere else the parser would discard `<x>`.
